# 浏览器基础

## 什么是进程？
- 一个程序需要自己的专属内存空间，这个内存空间可以简单理解为进程
- 每个应用至少有一个进程，相互独立，需要通信得双方同意

## 什么是线程？
- 一个进程至少有一个线程。
- 进程开启后自动创建一个线程来运行代码，该线程称为主线程。

## 浏览器有哪些进程和线程？
- 浏览器：多进程多线程
  - 浏览器进程
  - 网络进程：加载网络资源。
  - 渲染进程：每个标签页开启一个新渲染进程，相互不影响。

## 如何理解JS的异步
> 为什么？
> JS是单线程语言，运行在`浏览器的渲染主线程中`，渲染主线程只有一个。（渲染页面，执行JS）
> 使用同步可能会造成堵塞，页面无法更新，会造成卡死的现象，所以使用异步来解决。
> 所以
> 所以采用异步解决。具体做法：当某些任务发生时，主线程将任务交给其他线程去处理，自己立即结束任务的执行，转而执行后续代码。当其他线程完成时，将事先传递的回调函数包装成任务，加入到消息队列的末尾排队，等待主线程调度执行。
在这种异步模式下，浏览器永不阻塞，从而最大限度的保证了单线程的流畅运行。

## 什么是事件循环（消息循环）
事件循环又叫做消息循环，是浏览器渲染主线程的工作方式。
在Chrome的源码中，它开启一个不会结束的for循环，每次循环从消息队列中取出第一个任务执行，而其
他线程只需要在合适的时候将任务加入到队列末尾即可。
过去把消息队列简单分为宏队列和微队列，这种说法目前已无法满足复杂的浏览器环境，取而代之的是一种更
加灵活多变的处理方式。
根据W3C官方的解释，每个任务有不同的类型，同类型的任务必须在同一个队列，不同的任务可以属于不同的
队列。不同任务队列有不同的优先级，在一次事件循环中，由浏览器自行决定取哪一个队列的任务。但浏览器
必须有一个微队列，微队列的任务一定具有最高的优先级，必须优先调度执行。

### 第1步 解析HTML。（parse）
渲染的第一步是解析HTML。
解析过程中遇到CSS解析CSS，遇到JS执行JS。为了提高解析效率，浏览器在开始解析前，会启动一个预解析的线程，率先下载HTML中的外部CSS文件和外部的JS文件。

如果主线程解析到link位置，此时外部的CSS文件还没有下载解析好，主线程不会等待，继续解析后续的
HTML。这是因为下载和解析CSS的工作是在预解析线程中进行的。这就是CSS不会阻塞HTML解析的根本原因。

如果主线程解析到script位置，会停止解析HTML，转而等待JS文件下载好，并将全局代码解析执行完成后，才能继续解析HTML。这是因为JS代码的执行过程可能会修改当前的DOM树，所以DOM树的生成必须暂停。这就是JS会阻塞HTML解析的根本原因
第一步完成后，会得到DOM树和CSSOM树，浏览器的默认样式、内部样式、外部样式、行内样式均会包含
在CSSOM 树中。

### TCP 三次握手
```
客户端 ──── SYN ────→ 服务端
客户端 ←── SYN-ACK ←── 服务端
客户端 ──── ACK ────→ 服务端
```

### TLS 握手（HTTPS）

## 高级

### 垃圾回收（GC）

#### 算法
| 算法 | 说明 |
|------|------|
| 标记-清除 | 从根对象标记，清除未标记对象 |
| 引用计数 | 计数为 0 时回收 |
| 标记-整理 | 整理内存，解决碎片 |

#### V8 GC 策略
- **分代回收**：新生代 + 老生代
- **Scavenge**：新生代，快速
- **Mark-Compact**：老生代，整理内存
- **增量标记**：分批执行，避免卡顿

### 内存泄漏

#### 常见原因
| 原因 | 说明 |
|------|------|
| 全局变量 | 未声明的变量挂在 window |
| 闭包 | 持有外部变量引用 |
| 定时器 | setInterval 未清理 |
| 事件监听 | addEventListener 未移除 |
| DOM 引用 | 删除后变量仍引用 |

#### 排查与优化
- DevTools → Memory 快照对比
- Performance Monitor
- 及时清理定时器/监听
- 使用 WeakMap/WeakSet

|----------|------|
| 宏任务 | setTimeout, setInterval, I/O, UI 渲染 |
| 微任务 | Promise.then, MutationObserver, queueMicrotask |

**执行顺序**：主线程 → 微任务 → 宏任务

```js
async function test() {
  console.log(1);
  await Promise.resolve();
  console.log(2); // 微任务
}
test();
console.log(3);
// 输出：1, 3, 2
```

### Web Worker

#### 是什么
- 后台 JavaScript 线程
- 与主线程隔离，不阻塞 UI
- 无法访问 DOM、window

#### 类型
| 类型 | 说明 |
|------|------|
| Dedicated Worker | 专用，单页面使用 |
| Shared Worker | 共享，多页面共用 |
| Service Worker | PWA 缓存/离线（生命周期：install → activate → fetch → message） |

#### 通信
```js
// main.js
const worker = new Worker('worker.js');
worker.postMessage({ data: 100 });
worker.onmessage = e => console.log(e.data);

// worker.js
self.onmessage = e => self.postMessage(e.data * 2);
```

#### 场景
复杂计算、数据处理、加密解密、图像处理、文件压缩
